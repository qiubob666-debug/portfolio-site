# ADR-0001: 个人门户作为一切方案的展示中心

**日期:** 2026-04-23
**状态:** Accepted
**决策者:** Bob Qiushao
**层级:** Strategic
**关联 ADR:** 衍生出 ADR-0002 / ADR-0011 / ADR-0020

---

## 背景

Bob 拥有多项已实战验证的技术能力(认证社区 / Headless 矩阵 / 内容工厂 / 高审美前端等)。

之前的思路倾向于把这些能力**包装成几个固定产品**(如 "Auraloop Commerce / Growth / Auth"),但这种思路存在三个根本问题:

1. **限制了能力的扩展性** — 产品名一旦固定,新能力难以无缝加入
2. **限制了客群的多样性** — 产品包装天然偏向某一类客户(如电商),其他客群(工厂/品牌/个人)看不到自己的影子
3. **限制了 Bob 的成长性** — Bob 是一个持续学习的工程师,不是一个固定 SKU 的卖家

Bob 明确表态:

> "我相信我的学习能力和执行能力,第五、第六、第七将是我后续根据其他需求所搭建实现的。"

## 决策

**采用"个人门户"模型,而非"产品组合"模型。**

bobqiushao.online 不是一个产品销售页,而是 **Bob 个人能力的总展示中心**。

```
个人门户 (root)
    ↓
    展示我所有可复用的方案能力(开放数量,持续扩展)
    ↓
    每个能力背后是一个或多个实战案例(Auraloop 是首个 Live)
    ↓
    每种客群(跨境/工厂/品牌/个人)看到针对自己的落地页
    ↓
    用前缀子域名快速复制为该客群的专属落地页
    ↓
    成交后该客户也成为新案例,反哺能力库
```

## 备选方案

| 方案 | 优点 | 缺点 | 为什么没选 |
|---|---|---|---|
| A. 产品组合(Auraloop X/Y/Z) | 销售话术清晰、易复用 | 锁死能力边界、限制客群多样性 | 与 Bob 长期主义和成长性诉求冲突 |
| B. 纯接单个体户 | 简单,无需架构 | 无积累、无壁垒、不可扩展 | 与 Bob 的工程能力不匹配,资产无法沉淀 |
| C. **个人门户(选定)** | 开放、可扩展、能反映真实能力深度 | 销售话术需要更细致设计 | 唯一能容纳 Bob 长期野心的架构 |
| D. 公司化品牌(如 Auraloop Studio) | 有"机构感",可议价能力强 | 与"独立工程师"定位冲突,初期成本高 | 时机未到,可作为 12-24 个月后的演进方向 |

## 后果

### 正面
- 能力库可无限扩展,不受产品 SKU 限制
- 多客群可同时服务,不必"二选一"
- Bob 的学习曲线本身成为业务护城河
- 案例库 + 能力库 + 落地页形成三向复利

### 负面 / 已知风险
- 销售话术比"卖具体产品"更复杂,需要客户理解"能力"概念
- 主页信息架构难度更高,容易显得"什么都做"
- 客户决策周期可能更长(需要更多教育)
- 风险缓解:**用客群专属落地页**(子域名前缀方案)解决

### 触发重新评估的条件
- 当 Bob 决定团队化时(从一人 → 多人),可能需要转向更标准化的产品组合
- 当某个能力域产生超过 60% 营收时,可能需要将其独立成子品牌
- 每年 12 月强制审视一次,看模型是否还匹配业务现状

## 实施清单

- [ ] 主页信息架构按"个人门户"重构
- [ ] 主页同时呈现:个人介绍 / 能力库 / 案例库 / 客群入口
- [ ] 为每个客群建立专属子域名落地页(参见 ADR-0011)
- [ ] 在所有对外文案中,**不使用"产品名"包装能力**(如 "Auraloop Commerce")
- [ ] 把 Auraloop 作为"实战样板"而非"产品",参见 ADR-0003
