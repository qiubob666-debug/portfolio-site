## 一、世界观

### 1.1 我是谁

我是 Bob Qiushao。一个独立工程师 + 设计师 + 创业者,使用 AI 高强度协作的工作模式,正在构建一套**可复利的方案能力体系**。

### 1.2 我在做什么

我在做的不是"接建站单",而是构建一个**个人门户**:

```
bobqiushao.online (主站)
    ↓
    展示我所有可复用的方案能力
    ↓
    每个能力背后是一个可演示的实战样板(如 Auraloop)
    ↓
    每种客群(跨境/工厂/品牌/个人)看到针对自己的落地方案
    ↓
    用前缀子域名快速复制为该客群的专属落地页
    ↓
    成交后该客户也成为新案例,反哺能力库
```

**核心逻辑:** 一个能力 → 多个落地页 → 多个客群 → 多个案例 → 反哺能力 → 更多落地页

### 1.3 我的真实壁垒(已落地于 Auraloop 实战)

**不是"几个产品",而是"四类通用能力 + 持续学习的开放架构"。**

| # | 能力域 | Auraloop 样板的实现方式 | 通用化后能服务什么客群 |
|---|---|---|---|
| 1 | **认证 + 社区 + 询盘** | Google OAuth + Supabase + 邮件管理 | 任何需要海外用户体系/B2B 询盘的项目 |
| 2 | **品牌矩阵 / 多门户** | Headless WP+WC 后端 + Vercel 多前缀域名 + 零成本平替 Shopify | 跨境多品牌测品 / 工厂多产品线 / 品牌全球化 |
| 3 | **内容生产工作流** | n8n 工作流 + Sanity CMS | 任何需要持续 SEO 内容/多语言/多平台分发 |
| 4 | **高审美前端 + CDN/性能** | 现代设计语言 + Cloudflare/Vercel CDN | 提高任何项目的专业度和转化率 |
| 5+ | **(开放)** | 后续陆续补充:选品系统 / BI 数据 / ... | 按客户痛点扩展 |

**关键认知:**
- 这套能力 **不限于服务跨境电商** — 当前 bobqiushao.online 面向跨境只是第一个落地页,不是天花板
- 这套能力 **不限于挂在 Auraloop 品牌下** — Auraloop 是样板间,不是品牌包装
- 这套能力 **不限于 5 项** — 我相信我的学习能力,后续会持续扩展

### 1.4 我不会做的事

- ❌ 把自己绑死在"建站个体户"的标签上
- ❌ 用一个虚构的产品名(Auraloop XX / OS XX)去包装真实能力
- ❌ 一次性卖死客户,放弃年付/月付的复利现金流
- ❌ 因为短期成交压力,牺牲方案的通用性和可复用性
- ❌ 重做已具备的资产(6 个珠宝/手表案例 / 1688 渠道 / Auraloop 实战栈)

## 二、个人门户的真实定位

### 2.1 bobqiushao.online 不是终点,是入口

**当前形态(Phase 0):**
- 一个面向跨境电商卖家的销售页

**真实定位(Phase 1+):**
- 一个**个人能力门户**,展示我能为不同客群提供什么
- 通过前缀子域名,快速生成针对各客群的专属落地页

```
bobqiushao.online                    ← 主门户 / 关于我 / 全部能力
ecom.bobqiushao.online              ← 跨境电商客群专属落地页(=当前主页内容)
factory.bobqiushao.online           ← 工厂客群专属落地页
brand.bobqiushao.online             ← 品牌客群专属落地页
personal.bobqiushao.online          ← 个人客户专属落地页
case-{name}.bobqiushao.online       ← 每个案例独立子域名
auraloop.bobqiushao.online (或外链) ← Auraloop 样板间入口
```

### 2.2 个人主域名的策略

**当前域名:** bobqiushao.online(已有,继续使用)

**未来可考虑的更"个人门户感"的域名(可选,不强求):**
- `bob.work` / `bob.studio` / `bobqs.dev`
- `qiushao.io` / `qiushao.co`
- 任何短、品牌感强、易记的域名

**域名策略不是"换不换"问题,而是:**
- 主域名:个人品牌
- 子域名:能力 / 客群 / 案例
- 客户域名:他们自己买,我们帮他们挂载

**新增决策(用户原话):**
> "用前缀子域名为客户挂载项目,客户自己选前缀。"

### 2.3 从 Auraloop 反哺到主门户

Auraloop 已经实战验证的所有东西,**都应该:**
1. 提取出方法论 → 沉淀到 ADR
2. 抽象为可复用模板 → 沉淀到 capability-library
3. 包装成对外案例 → 展示在 bobqiushao.online

**Auraloop 是"我做过什么的最强证据",不是"我要卖的产品"。**

## 三、核心工作原则

### 3.1 长期主义优先

任何决策面临"快速成交 vs 可复用资产"二选一时:

**默认选可复用资产**,除非现金流真的告急。

理由:Bob 的差异化不是"做单速度",而是"方案体系的累积深度"。

### 3.2 AI 协作的边界

| 场景 | 由谁主导 | 谁辅助 |
|---|---|---|
| 战略决策 | **Bob** | Claude(深度对话伙伴) |
| 工程执行 | **Manus** | Bob 验收 |
| 编码实现 | **Cursor + Claude Code** | Bob 审核 PR |
| 内容创作 | n8n + Claude | Bob 终审 |
| 客户沟通 | **Bob** | AI 不直接对客户 |

**铁律:** AI 永远不直接代表 Bob 与真实客户沟通。AI 可以起草内容,Bob 必须人工审核后发出。

### 3.3 一切决策必须留痕

任何会影响未来的决策,**必须**写一份 ADR(Architecture Decision Record)。

不写 ADR 的决策视为**未做出**。

### 3.4 一切操作必须可重复

任何手动操作三次以上,**必须**沉淀为 RUNBOOK 或脚本。

无 RUNBOOK 的操作视为**不可扩展**。

### 3.5 一切产出必须可验收

任何任务交付,**必须**带验收清单。

无验收清单的交付视为**未完成**。

## 四、复利模型

### 4.1 资产层级

```
Layer 5: 客户实战 (CASE)
    ↑ 每个真实案例反哺到下层
Layer 4: 落地页 / 销售物料 (LANDING)
    ↑ 多客群专属落地页,基于能力库批量生成
Layer 3: 能力库 (CAPABILITY)
    ↑ 4 项已实现 + N 项可扩展
Layer 2: 工程基础 (FOUNDATION)
    ↑ ADR / RUNBOOK / CI/CD / 模板库
Layer 1: 学习与吸收 (INPUT)
    ↑ 顶级网站灵感 / UI 素材 / 技术案例 / 新工具
```

**核心特征:** 上层每发生一次,下层资产**自动增加一份**。

### 4.2 一年后的预期资产

| 层级 | 当前 (Apr 2026) | 1 年后目标 |
|---|---|---|
| Layer 1 学习 | 持续中 | 200+ inspiration / 50+ 技术案例 |
| Layer 2 工程 | 部分(Auraloop Sync) | 完整 ADR 体系 + 30+ RUNBOOK + 全自动 CI/CD |
| Layer 3 能力 | 4 项已实战 | 8-10 项可演示 |
| Layer 4 落地页 | 1 个(跨境) | 5-8 个(各客群) |
| Layer 5 案例 | 6 个珠宝手表 + Harrow + Auraloop | 20-30 个 |

## 五、给所有 AI Agent 的命令

### 5.1 你必须做的

1. **先读 Charter**(本文件),再读 ADR,再读 RUNBOOK,再开始任务
2. 任何不确定的决策,**问 Bob,不要猜**
3. 任何产出必须能映射到**长期资产层**(不只是短期任务)
4. 任何完成必须按 **验收清单**提交

### 5.2 你不能做的

1. ❌ 用产品名包装真实能力(不要叫 Auraloop X / OS X)
2. ❌ 把 Bob 的能力叙事限制在"建站""跨境"等狭窄标签
3. ❌ 假定 Bob 的能力库只有 4 项(它是开放的)
4. ❌ 直接修改 ADR(只能新增,不能改写)
5. ❌ 跳过验收清单交付任何产出

### 5.3 当你不确定时

按这个顺序找答案:

```
1. 当前任务的具体指令
       ↓ 没有时
2. 该任务所属阶段的 RUNBOOK
       ↓ 没有时
3. 相关的 ADR
       ↓ 没有时
4. 本 Charter
       ↓ 还是不确定
5. 停下来问 Bob
```
